# Migration: 0.2.0 To 0.3.0

## Summary

Renames the project from Signal Methodology to Signal Method and changes slugged identifiers from `signal-methodology` to `signal-method`.

This is a breaking rename intended for the pre-publication phase.

## Detect

A project is probably on 0.2.0 if it has any of these old identifiers:

- `signal-methodology.json` at the project root
- `$signal-methodology` in agent guidance
- `skills/signal-methodology/` in a local methodology source checkout
- `signal-methodology` in package or workspace metadata

## Steps

1. Rename the root manifest from `signal-methodology.json` to `signal-method.json`.
2. In the manifest, set `methodology` to `signal-method`.
3. Update project agent guidance from `$signal-methodology` to `$signal-method`.
4. Update documentation references from Signal Methodology to Signal Method where they name the product.
5. Rename local methodology source paths from `skills/signal-methodology/` to `skills/signal-method/` if the project vendors the skill source.
6. Update bootstrap or validation scripts that reference the old manifest or skill path.
7. Update workflow state with this migration's completed steps and next step.
8. Set `signal-method.json` to `0.3.0`.

## Do Not

- Rename the docs root as part of this migration.
- Rewrite canonical project docs unless they contain old product or manifest references.
- Keep both manifests unless a project needs a temporary compatibility bridge.

## Completion Checks

- The project root has `signal-method.json`.
- No active project guidance references `$signal-methodology`.
- No active bootstrap or validation path references `signal-methodology.json`.
- The recorded methodology version is `0.3.0`.
