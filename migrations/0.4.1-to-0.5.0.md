# Migration: 0.4.1 To 0.5.0

## Summary

Changes `workflow-state/current.md` from "active or most recent workflow state" into a permanent workflow-state index.

Every meaningful workflow now gets its own sibling state file under `workflow-state/`, even when only one workflow is active.

## Detect

A project is probably on 0.4.1 if:

- `signal-method.json` records version `0.4.1`
- `workflow-state/current.md` contains full workflow details such as `Goal`, `Completed`, `Current Step`, `Next Step`, and `Resume Instructions`
- there are no sibling workflow state files for active work
- doc-loading guidance says to read `current.md` "or the active workflow state file"

## Steps

1. Read the project's `signal-method.json` and confirm the configured docs root.
2. Update workflow state to record the methodology upgrade.
3. Create a sibling workflow state file for the active or most recent workflow, using a clear slug such as `workflow-state/methodology-upgrade-0.5.0.md`, `workflow-state/greenfield-initialization.md`, or a feature-specific name.
4. Move detailed workflow content from `workflow-state/current.md` into the sibling state file.
5. Rewrite `workflow-state/current.md` as a workflow index with:
   - purpose and last-updated date
   - active workflow list
   - link to each sibling workflow state file
   - short status and next step for each workflow
   - cross-workflow blockers or coordination notes
   - recently closed workflows when useful for handoff
6. Update `doc-index.md` if it describes `workflow-state/current.md` as the active or most recent workflow state instead of an index.
7. Update agent guidance or doc-loading guidance so agents read `workflow-state/current.md` first, then the sibling state file named by the index.
8. Set `signal-method.json` to `0.5.0` after the migration is complete.
9. Update the detailed workflow state and `current.md` index entry with the migration result and next step.

## Do Not

- Keep detailed per-workflow progress in `workflow-state/current.md`.
- Delete old workflow-state details before copying them into a sibling file.
- Rename the configured docs root unless the project separately requested that change.
- Overwrite project-specific workflow wording with the template when a smaller edit preserves current truth.

## Completion Checks

- `workflow-state/current.md` is an index, not a detailed workflow state file.
- Each active workflow has a sibling state file.
- The index links to every active workflow state file.
- The detailed state file records the migration result and next step.
- The recorded methodology version is `0.5.0`.
